# BT Beyond Manus / Opus Agent Blueprint

> 目标：不复制外部产品，不套壳外部仓库。只吸收一线产品暴露出的能力方向，重写成 BT（黑光）自己的本地优先高定代理人体系。

## 外部信号抽象

- Manus 类产品证明了用户要的是“一句话交付完整任务”，但真正稀缺的是可验证执行、可恢复长任务、清晰权限边界和产物审计。
- OpenAI Agents SDK 的新方向是把 agents 从胶水代码推进到带文件、工具、沙箱、追踪和长期运行基础设施的执行环境。
- Claude Opus 4.7 的方向是更强的长程任务、工具容错、文件系统记忆、高分辨率视觉和更严谨的自检。
- Anthropic Computer Use 的方向是受限虚拟机/容器、截图、鼠标键盘、agent loop，以及登录/提示注入场景的安全隔离。
- LangGraph / Deep Agents / 企业 agent 平台的方向是可恢复图工作流、部署观测、多人/多角色并行，以及把任务执行变成可管理系统。
- Hermes / Bottega / MetaHuman 级数字人方向不是炫技，而是材质、留白、皮肤体积、眼神高光、状态呼吸和高定秩序。

## 黑光自己的二次创造

BT 不做“另一个 Manus”。BT 做一个本地高定 Agent Control Plane：

- 用户只看见一个克制、真实、有气质的数字代理人。
- 系统内部是多角色代理团队：Planner、Builder、Verifier、Reviewer、Archivist、Safety Gate。
- 每一次行动都有证据链：目标、计划、工具调用、文件差异、验证命令、失败原因、下一步。
- 长任务可暂停、可恢复、可回看，不靠聊天记录硬撑上下文。
- 高风险动作必须经过确认；低风险维护动作可以自动完成并留下审计。
- 模型不是品牌崇拜，而是路由策略：本地 Omni 处理本机/多模态，强推理模型处理复杂设计与代码审查，小模型做路由和摘要。

## 能力分层

### P0：Doctor 360

- 汇总 Git、进程、端口、前后端健康、模型网关、环境体积、最近失败日志。
- 所有“完成了”的回答必须能挂上真实验证证据。
- `/meta/doctor` 输出给前端和 Agent 共用，避免 UI 与真实运行状态脱节。

### P1：Run Graph

- 把一次任务拆成 `observe -> plan -> patch -> verify -> report` 节点。
- 每个节点记录输入、产出、命令、风险等级和失败恢复建议。
- 前端用时间线和状态卡展示，不做花哨噪声。

### P2：Skill Forge

- Agent 发现重复任务时，只生成“候选技能”，不直接写长期记忆。
- 用户确认后再固化为技能包，避免记忆污染。
- 技能必须包含触发条件、禁用条件、验证命令和回滚风险。

### P3：Computer-Use Sandbox

- 只在隔离桌面/浏览器/容器里做自动点击、截图和键盘输入。
- 登录、付款、发布、权限变更、删除和外网下载默认需要用户确认。
- 每一步截图或 DOM 证据进入执行记录，失败后可复盘。

### P4：Expert Council

- Planner 负责拆任务和边界。
- Builder 负责最小可用补丁。
- Verifier 负责真实命令和反例。
- Reviewer 负责找回归、隐私、权限、生成物入库风险。
- Archivist 负责总结可复用经验，但只写候选记忆。

### P5：Maison Digital Human

- 视觉标准：真人脸优先，低饱和金棕、皮革棕、丝绸象牙、铜色微光。
- 动效标准：呼吸、视线、轻微景深、低频光影；不用赛博扫描线和粒子暴雨。
- 状态标准：静候、聆听、思考、执行、验证、完成、需要确认。
- 技术标准：`photo.png + depth.png + manifest.json`，可替换素材，不把用户照片进 Git。

### P6：Opus-Level Harness

- 对复杂任务启用更高推理预算，但必须有任务预算和停止条件。
- 每次输出前跑“自证清单”：改了什么、证据在哪、风险是什么、还没做什么。
- 对代码类任务启用二阶段检查：实现者补丁 + 审查者找错。

### P7：Artifact Vault

- 把报告、截图、构建日志、验证结果和关键 diff 存成可查证产物。
- 生成媒体、本地照片、模型权重、`.env`、数据库和日志默认不进 Git。
- 可发布内容和私密运行内容分仓/分区管理。

## Cursor / Agent 执行提示

当用户要求“最高端、超越 Manus、比肩 Opus”时，不要马上堆功能。按下面顺序执行：

1. 先读 `AGENTS.md`、`CLAUDE.md`、本蓝图和当前 `git status`。
2. 判断改动属于 Doctor、Run Graph、Skill Forge、Sandbox、Council、Digital Human 哪一层。
3. 只做当前层最小但真实可验证的补丁。
4. 不引入外部仓库实现；如需参考，只写设计原则和验收标准。
5. 跑真实验证；无法验证时明确说原因。
6. 把可合并代码和私密/生成物分开，避免把照片、深度图、日志、模型文件入库。

## 下一阶段验收标准

- 前端能显示数字人状态和任务执行节点，不只是聊天气泡。
- 后端能给出一次任务的结构化运行图和验证证据。
- `/meta/doctor` 能识别坏环境、缺模型、端口异常和生成物入库风险。
- Cursor/Claude/Codex 都能读到同一套高端优化规则。
- Git 暂存区只包含可审查代码、文档和安全模板，不含隐私照片或大模型产物。
