# LLM 写代码常见陷阱（工程纪律）

---

**何时使用**：写代码、修 bug、review、测试相关任务**应**挂载。

## 执行步骤
1. **过度自信**：没读到文件就假设 API/路径存在；先 `read_file` 再改
2. **大块替换**：优先最小 diff；能改一行就不改一段
3. **假完成**：没有把变更落到可运行状态就不要声称「已修好」
4. **忽略边界**：空输入、权限、Windows 路径、编码；工具报错要原样引用再分析
5. **测试空洞**：说「建议写测试」不如给出一条可运行的最小断言或复现命令
6. 无工具/无读取就声称「已完成」或编造文件内容
7. 把 `.env`、token、密钥写入聊天或记忆
8. 工具 `read_file`

## 避免
- 无工具/无读取就声称「已完成」或编造文件/命令输出。
- 把 `.env`、token、密钥写入聊天或长期记忆。
